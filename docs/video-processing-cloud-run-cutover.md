## Prereqs

- Platform already provisioned (GCP resources exist; tofu outputs are available via `PLATFORM_STATE_R2_*`).

## Steps

### 1. Deploy the Cloud Run service (first time)

Trigger the workflow manually:

```bash
gh workflow run deploy-video-worker.yaml -R matthewtapps/syllabus-tracker
```

The workflow will:
1. Authenticate to GCP via WIF (no stored credential).
2. Build the container from `crates/video-worker/Dockerfile` (context = repo root) and push to Artifact Registry as `australia-southeast1-docker.pkg.dev/matthewtapps-sillybus/video-worker/transcode-server:<sha>` + `:latest`.
3. Read `ENQUEUE_TOKEN`, `VIDEO_CALLBACK_SECRET`, `S3_ACCESS_KEY`, `S3_SECRET_KEY` from platform tofu state (via the existing `PLATFORM_STATE_R2_*` bootstrap creds).
4. Deploy `transcode-server` Cloud Run service (public, bearer-auth at the app level, `--no-cpu-throttling`, 2 Gi RAM, 600s timeout, max 3 instances).
5. Print the service URL to the step summary.

Note the URL from the summary (e.g. `https://transcode-server-<hash>-ts.a.run.app`).

### 2. Set repo variables

```bash
gh variable set VIDEO_ENQUEUE_URL \
  -b "https://transcode-server-<hash>-ts.a.run.app/jobs" \
  -R matthewtapps/syllabus-tracker

gh variable set VIDEO_PROCESSOR_STAGING \
  -b "cloudflare" \
  -R matthewtapps/syllabus-tracker
```

(`VIDEO_PROCESSOR_STAGING=cloudflare` is the existing app-level feature flag; the value selects the `CloudflareProcessor` code path, which now POSTs to the Cloud Run URL.)

### 3. Re-run staging deploy

```bash
gh workflow run staging.yml \
  -f branch=roadmap/threads-08-video-processing \
  -f refresh_db=false \
  -f allow_destructive_migrations=false \
  -R matthewtapps/syllabus-tracker
```

Staging will receive the updated `VIDEO_ENQUEUE_URL` and route video jobs to Cloud Run.

### 4. Validate

- Upload a video at https://staging.sillybus.app.
- Confirm the row transitions `processing -> ready` and playback works (object served from R2).
- Check Cloud Run logs in GCP console (project `matthewtapps-sillybus`, region `australia-southeast1`, service `transcode-server`) — expect a transcode request log per upload.
- Confirm the web droplet CPU stays low during uploads (transcoding is off-host).
- No rows stuck in `processing` past `VIDEO_PROCESSING_TIMEOUT_SECONDS` (swept by the background sweeper).

### 5. Roll back (if needed)

```bash
gh variable set VIDEO_PROCESSOR_STAGING -b "host" -R matthewtapps/syllabus-tracker
gh workflow run staging.yml \
  -f branch=roadmap/threads-08-video-processing \
  -f refresh_db=false \
  -f allow_destructive_migrations=false \
  -R matthewtapps/syllabus-tracker
```

Host transcoding resumes immediately.

## Notes

- **No Cloudflare secrets needed.** The old `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID` references are gone. The `ENQUEUE_TOKEN` and `CALLBACK_SECRET` values are reused as generic bearer/HMAC credentials; they were generated by platform tofu and are provider-agnostic.
- **Leftover Cloudflare Queue** (`sillybus-video-jobs` + DLQ) and any CF queue tofu resources in `../infra` are now inert (nothing enqueues to them). Removing them is a separate follow-up infra PR; they cost nothing while idle.
- **CPU billing:** `--no-cpu-throttling` is set so the synchronous transcode request keeps CPU for its duration. At low volume this is well within the Cloud Run free tier (180,000 vCPU-seconds/month).
- **Prod cutover** is intentionally NOT covered here; it is a separate, explicit future step.
