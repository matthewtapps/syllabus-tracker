## Steps

### 1. Deploy the Cloud Run service (first time)

Trigger the workflow manually:

```bash
gh workflow run deploy-video-worker.yaml -R matthewtapps/syllabus-tracker
```

The workflow will:
1. Authenticate to GCP via WIF (no stored credential).
2. Build the container from `crates/video-worker/Dockerfile` (context = repo root) and push to Artifact Registry as `australia-southeast1-docker.pkg.dev/matthewtapps-sillybus/video-worker/transcode-server:<sha>` + `:latest`.
3. Read `ENQUEUE_TOKEN`, `VIDEO_CALLBACK_SECRET`, `S3_ACCESS_KEY`, `S3_SECRET_KEY` from platform tofu state (via the existing `PLATFORM_STATE_R2_*` bootstrap creds).
4. Deploy `transcode-server` Cloud Run service (public, bearer-auth at the app level, `--no-cpu-throttling`, 2 Gi RAM, 600s timeout, max 3 instances).
5. Print the service URL to the step summary.

Note the URL from the summary (e.g. `https://transcode-server-<hash>-ts.a.run.app`).

### 2. Set repo variables

```bash
gh variable set VIDEO_ENQUEUE_URL \
  -b "https://transcode-server-<hash>-ts.a.run.app/jobs" \
  -R matthewtapps/syllabus-tracker

gh variable set VIDEO_PROCESSOR_STAGING \
  -b "remote" \
  -R matthewtapps/syllabus-tracker
```

(`VIDEO_PROCESSOR_STAGING=remote` selects the `RemoteProcessor` code path, which POSTs to the Cloud Run URL.)

### 3. Re-run staging deploy

```bash
gh workflow run staging.yml \
  -f branch=roadmap/threads-08-video-processing \
  -f refresh_db=false \
  -f allow_destructive_migrations=false \
  -R matthewtapps/syllabus-tracker
```

Staging will receive the updated `VIDEO_ENQUEUE_URL` and route video jobs to Cloud Run.

### 4. Validate

- Upload a video at https://staging.sillybus.app.
- Confirm the row transitions `processing -> ready` and playback works (object served from R2).
- Check Cloud Run logs in GCP console (project `matthewtapps-sillybus`, region `australia-southeast1`, service `transcode-server`) — expect a transcode request log per upload.
- Confirm the web droplet CPU stays low during uploads (transcoding is off-host).
- No rows stuck in `processing` past `VIDEO_PROCESSING_TIMEOUT_SECONDS` (swept by the background sweeper).

### 5. Roll back (if needed)

```bash
gh variable set VIDEO_PROCESSOR_STAGING -b "host" -R matthewtapps/syllabus-tracker
gh workflow run staging.yml \
  -f branch=roadmap/threads-08-video-processing \
  -f refresh_db=false \
  -f allow_destructive_migrations=false \
  -R matthewtapps/syllabus-tracker
```

Host transcoding resumes immediately.

## Notes

- **No Cloudflare secrets needed.** The old `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID` references are gone. The `ENQUEUE_TOKEN` and `CALLBACK_SECRET` values are generic bearer/HMAC credentials; they were generated by platform tofu and are provider-agnostic.
- **Leftover Cloudflare Queue** (`sillybus-video-jobs` + DLQ) is being removed in infra PR `drop-cf-queue` (the secrets stay). Nothing enqueued to it.
- **CPU billing:** `--no-cpu-throttling` is set so the synchronous transcode request keeps CPU for its duration. Comfortably within the Cloud Run free tier for expected volume (~250-500 four-min videos/month free; baseline ~150 = $0). Memory (2 GiB) is the binding free-tier limit; drop to 1 GiB for ~2x headroom if ever needed.

**Staging validated 2026-06-15:** a real upload processed end-to-end on Cloud Run (`processing -> ready`, played from R2).

## DEFERRED: when merging the PR stack to main

The whole roadmap stack (PRs #51 -> #62) merges to main by merging the **top** (#62). Two follow-ups belong with that merge:

1. **Re-enable the `deploy-video-worker` push-trigger.** It is currently `workflow_dispatch`-only with the `push` block commented out (because `workflow_dispatch` requires the workflow to exist on the default branch, which it didn't pre-merge — we fired the first deploy via a temporary branch push-trigger). Once the workflow is on `main`, uncomment the `push:` block in `.github/workflows/deploy-video-worker.yaml` (branches: `[main]`, paths: the video crates) so worker/image changes auto-deploy to Cloud Run. The temporary-branch-trigger hack is then no longer needed.
2. Confirm the infra `drop-cf-queue` PR is merged (removes the inert CF queue).

Prod is NOT cut over by the merge: `config/common.env` defaults `VIDEO_PROCESSOR=host`, and `config/prod.env` does not override it, so the prod app keeps host transcoding until the explicit cutover below.

## DEFERRED: prod cutover (separate, explicit step)

The Cloud Run service is account-level and already serves both envs (it posts back to whatever `callback_url` each job carries). So prod needs **app-deploy wiring only — no new GCP infra**. Mirror what `staging.yml` does, in the prod path (`deploy.yaml`):

1. In `deploy.yaml`, add a step that reads `_platform_video_enqueue_token` + `_platform_video_callback_secret` from platform tofu state and ships them in the prod runtime-secrets bundle (exactly like the honeycomb keys; same as the block added to `staging.yml`).
2. Set the prod app env: `VIDEO_PROCESSOR=remote`, `VIDEO_ENQUEUE_URL=<cloud-run-url>/jobs`, `VIDEO_CALLBACK_BASE_URL=https://sillybus.app`. Put these in the prod deploy (e.g. `config/prod.env` for the non-secret ones, or the deploy env), NOT `config/common.env` (which is shared).
3. Deploy main. Upload a prod video, confirm `processing -> ready` + playback + low droplet CPU.
4. **Rollback:** set prod `VIDEO_PROCESSOR=host` and redeploy. Host transcoding resumes instantly (the seam + host path are retained).

Optional before prod: raise Cloud Run `--max-instances` if prod upload concurrency warrants it (staging uses 3).
