# Cross-repo review: feat/sillybus-app + ../infra (2026-06-08)

Reviewer: Claude. Scope: every file changed on `feat/sillybus-app` vs
`main`, plus the entire `../infra` repo. Focus: conventions,
extensibility, platform/service seam, project structure, workflow
currency, documentation, comment quality.

Findings are prioritised. **Blocker** = ship-blocking. **Should-fix**
= worth doing before this branch lands or shortly after.
**Nit** = preference / cleanup. **Strength** = working well, keep it.

---

## Overall verdict

The architecture is the strongest part. The platform/service split is
clean and the "Path 3" model (service CI consumes platform tfstate via
one bootstrap R2 token, derives everything else dynamically) is doing
exactly what you'd want: one manual cross-repo handoff, then the rest
of the credentials, hostnames, and ingest keys flow per-run from
state. Adding a second service or a sibling instance is plausibly a
small commit each, not a refactor.

The weakest part is **documentation hygiene**. Several docs in
`../infra/docs/` were written as forward-looking design or
session-of-the-moment status logs and have not been retired or
condensed now that the work landed. Anyone reading the repo cold today
will trip on stale plans that describe state that no longer exists.

Workflow design is solid (the four-job split in `deploy-sillybus.yaml`
is a meaningful gate, not ceremony), but the workflows themselves
contain considerable copy-paste that should be extracted before adding
a third one.

No security blockers spotted. No correctness blockers spotted in the
IaC. The biggest correctness risk is operational, not structural:
several places carry incident-driven workarounds (cloud-init module
trimming, sshd-keygen restart, AWS_MAX_ATTEMPTS, manual ssh-keyscan)
that are individually reasonable but together represent a fair amount
of "tribal knowledge" baked into config. Comments explain most of it
well; one or two need expansion (called out below).

## Strengths (keep doing)

1. **Platform/service seam.** `data "terraform_remote_state"
   "platform"` in `infra-sillybus/data.tf:6` is the only coupling
   point. Every credential, IP, zone id, and ingest key the service
   needs is re-exported via underscore-prefixed outputs
   (`infra-sillybus/outputs.tf:8` onwards). The convention
   "_platform_*" is self-documenting: "this is read from platform
   state, not minted here." That makes the seam visually obvious in
   any file that uses these values.

2. **Declarative delegation map.** `infra/tofu/credentials_delegations.tf:14`
   adding a new service is genuinely an entry in the `services` map.
   The IAM, R2 token, Honeycomb keys, bootstrap token all derive
   automatically via `for_each`. Compare to the alternative of
   per-service .tf files; this scales linearly to N services with no
   structural change.

3. **Generic host config.** `infra/nixos/hosts/sugar-glider/configuration.nix`
   knows nothing about sillybus by name. Adding a second service to
   the same host is a service-repo commit only (assuming the new
   service follows the same `*.enc.env` + `traefik-dynamic/<svc>.yml`
   + `/srv/<svc>/` contract). The NOPASSWD sudo rules
   (`configuration.nix:201-231`) are pattern-based and require no
   nixos change per service. This is the right shape.

4. **Cattle pattern with custom image.** The qcow-image-from-flake
   path (`build-host-image.yaml`) lets you boot 1GB droplets that
   kexec couldn't survive, and host destroy+recreate becomes a
   `var.compute_hosts.<name>.image` bump. The lifecycle stance
   ("`image` is NOT ignored; changing triggers destroy+recreate" at
   `compute_hosts.tf:78`) is correct and well-commented.

5. **Cloudflare runtime token derivation.** `cf_runtime_r2_tokens` in
   platform state holds (access_key_id, sha256(secret)) pairs the
   service can consume directly as AWS_*-shaped env vars. The
   comment chain explaining why S3-compat needs sha256-derived keys
   (`outputs.tf:128-147`) is exactly the kind of "why" that future
   you will be grateful for.

6. **Honeycomb provider IaC.** The full pipeline (Management Key →
   per-host ingest, per-service backend/browser ingest with
   `create_datasets` lock for the browser key) is IaC end-to-end.
   `credentials_honeycomb.tf:69-82` is a clean way to express the
   asymmetric trust model. Rotation is a single `tofu apply -replace`
   plus a redeploy with zero source edits.

7. **Litestream-restore as a workflow stage.** Moving restore out of
   the compose stack into its own GHA job (`deploy-sillybus.yaml:349`)
   is correct: a restore failure halts the deploy explicitly instead
   of leaving you with a partially-up stack and an ambiguous health
   check. The shell wrapper that distinguishes "no replica" from
   "actual restore failure" (`deploy-sillybus.yaml:393-407`) is the
   kind of nuance that's hard to get right inside compose.

8. **`.sops-sillybus.yaml` separate from legacy `.sops.yaml`.** Lets
   the legacy main branch keep its own SOPS config without conflict.
   Cleanly anticipates the post-cutover deletion of the legacy
   files. The named anchor (`&sugar_glider_host`) makes the
   rotation-after-host-rebuild edit a one-line change.

9. **Drift-guard on `nixos/ci-deploy.pub`.** `tofu-apply.yaml:47-62`
   catches the most likely operator error in this setup (laptop apply
   produces a new pubkey, dev forgets to commit). Failing CI early
   with a clear message is the right ergonomic.

10. **`prep_host` → `restore_litestream` → `migrate` → `deploy` split.**
    `deploy-sillybus.yaml:235-548`. The dependency graph is precise:
    each job has the *minimum* it needs, failures gate downstream
    work cleanly, re-running an individual failed job is meaningful.

### S2. Workflow duplication

`deploy-sillybus.yaml` has six jobs (`read_platform`, `apply_tofu`,
`prep_host`, `restore_litestream`, `migrate`, `deploy`) that each
repeat the same ~25 lines of setup: checkout, install-nix-action,
install opentofu, stage age key (sometimes), `tofu init` with the
30-attempt retry loop, pull SSH key, `ssh-keyscan`, etc.

Plus `bootstrap-sibling.yaml` duplicates a similar setup, slightly
differently (no retry loop on `tofu init`, sops via curl rather than
nix profile, etc.).

Plus `../infra/.github/workflows/tofu-apply.yaml`,
`nixos-deploy.yaml`, `build-host-image.yaml`, `state-backup.yaml` all
re-do the install-nix-action + nix-install-tofu + stage-age-key
dance.

That's ten copies of nearly the same setup. Options:

1. **Composite action (recommended).** Put the setup in
   `.github/actions/platform-state-reader/action.yaml` (per-repo)
   with inputs `aws-access-key-id`, `aws-secret-access-key`. It
   handles install + retry + `tofu init` + SSH key extraction +
   `ssh-keyscan` and emits outputs `vm_ip`, `host_age_recipient`,
   `ssh_key_path`. Then each job becomes `uses:
   ./.github/actions/platform-state-reader` + one or two job-specific
   steps.

2. **Reusable workflow.** Same idea via
   `workflow_call`-style reuse. Slightly heavier; only worth it for
   whole-job sharing across repos. The composite action is the right
   fit here.

The tofu-init-retry loop in particular is the same 8 lines in 8
different files; if the github CDN ever changes its 5xx behavior
again, all eight have to change. Pull this out.

### S3. `bootstrap-sibling.yaml` is partially diverged from
`deploy-sillybus.yaml`

The sibling workflow shares most of the deploy logic but with
gratuitous differences:

- `tofu init` without retry (`bootstrap-sibling.yaml:122,168`) — the
  same CDN that needed 30 attempts in deploy can return 5xx here too
- inline `curl` install of sops (`bootstrap-sibling.yaml:155-159`)
  instead of `nix profile install nixpkgs#sops`
- a generated `docker-compose.override.yml`
  (`bootstrap-sibling.yaml:264-296`) that duplicates four of the five
  compose service definitions just to override `env_file:` paths

The override file in particular is fragile: every time
`docker-compose.nixos.yml` gains a service or a service gains an
`env_file:` entry, the sibling override falls out of sync silently
(the sibling won't have the new env var; only by reading both files
will you spot the drift).

Two suggestions:
- Refactor the runtime stack to source secrets from a *single*
  `env_file:` whose path is parameterized (e.g.
  `${RUNTIME_SECRETS_FILE:-/run/runtime-secrets/sillybus.env}`).
  Then sibling sets `RUNTIME_SECRETS_FILE` and no override file is
  needed. Two pre-merged env files become one if you concatenate
  sillybus.env + sillybus-honeycomb.env at decrypt time, or you keep
  both and accept two parameterized vars.
- Or fold sibling and primary into one workflow with an input
  `instance_name` that defaults to `sillybus`; the only real
  differences are project name, host port, and "do you restore from
  X bucket". The dual-workflow approach is a copy that will diverge.

### S4. `infra/docs/honeycomb-telemetry.md` and `infra/docs/multi-provider.md` are essays

Both are long (642 and 364 lines), well-written, and contain real
thinking. But they're at a different level than the rest of the docs
(which are runbooks/reference). Consider:

- moving each into a top-level `infra/design/` dir with a brief
  README noting "these are design memos; the implementations live
  in tofu/ and nixos/"; OR
- ruthlessly shrinking. `multi-provider.md`'s pricing matrix and
  per-provider gotchas are useful; the "Why bother" intro could be
  three bullets.

Not blocking; doc bloat is the only real downside. The "Sources"
sections at the bottom of both are good.

### S5. `infra-sillybus/outputs.tf` underscore convention is undocumented

The `_platform_*` prefix convention shows up nowhere outside the file
itself. Anyone modifying the service tofu later will not know why
some outputs have leading underscores and some don't (in fact all
current outputs do; the convention is *all outputs are re-exports*,
which means the underscore is doing no work).

Either:
- drop the underscores (no resources in this module own outputs
  that conflict), and let the file's top-of-file comment explain
  "these are all re-exports from platform state"; OR
- write the convention down in the file's top-of-file comment
  ("`_platform_*` prefix = re-exported from
  data.terraform_remote_state.platform; non-prefixed outputs would be
  service-owned, none exist today") so it's discoverable.

Personally: drop the underscores. The convention isn't earning its
keep when 100% of outputs follow it.

### S6. `nginx/default.conf` has stale `otel_service_name`

`nginx/default.conf:5` sets `otel_service_name nginx-syllabus-tracker;`
but `docs/ai-telemetry-access.md:100` (in the same repo) and
`infra/docs/honeycomb-telemetry.md` consistently call this
`nginx-sillybus`. The naming inconsistency means Honeycomb queries
that filter on `service.name = nginx-sillybus` will silently miss
data. Rename the nginx OTel service to `nginx-sillybus` on this
branch.

### S7. `runtime-secrets-sillybus.enc.env` still contains
S3/Litestream creds that platform tofu mints

The runtime-secrets file holds `S3_ACCESS_KEY`, `S3_SECRET_KEY`,
`LITESTREAM_ACCESS_KEY_ID`, `LITESTREAM_SECRET_ACCESS_KEY`. But these
are exactly what `cf_runtime_r2_tokens["sillybus"]` re-export from
platform state (the same R2 runtime token has write to both
sillybus-videos-prod and sillybus-db-backups-prod). Right now,
rotating the R2 token = `tofu apply -replace` + manual
sops-decrypt + paste + sops-encrypt + commit + push. Compared to
the Honeycomb keys, which rotate via `tofu apply -replace` + redeploy
with zero source edits.

For consistency, the deploy workflow should also build a
`sillybus-runtime.enc.env` from
`_platform_runtime_r2_access_key_id` + secret at deploy time,
encrypted to the host recipient, the same way it builds
`sillybus-honeycomb.enc.env`. Then `runtime-secrets-sillybus.enc.env`
in-repo holds only the values that truly don't come from tofu
(`ROCKET_SECRET_KEY` — and even that could be a tofu-managed
random_string if you wanted; nothing forces it to be a secret you
chose).

This is the same pattern, just applied to a second set of values.
Worth doing before adding a second service that would otherwise
re-discover the same need.

### S8. `bootstrap-pipeline.md` references a workflow that doesn't exist

`infra/docs/bootstrap-pipeline.md:117-122` ends with `(TODO: write
.github/workflows/bootstrap-host.yaml ...)`. The actual current
bootstrap path is the qcow image build (`build-host-image.yaml`) +
`tofu apply` + `nixos-deploy.yaml`; nixos-anywhere isn't used at all
on the cattle path. Either delete the TODO (nixos-anywhere is the
fallback for the rare image-build-broken case) or commit to writing
the workflow. Don't leave a stale TODO in a runbook.

## Convention/structure notes

- **Tofu flat-file layout (prefix-based, no subdirs)**: works fine
  for a module this size. Matches the convention in some Terraform
  shops; others use subdirs. Either is conventional. The
  inconsistency is between the docs (which describe a subdirectory
  layout) and the code (which is flat) — fix the docs.

- **NixOS flake layout (modules/ + hosts/)**: matches the
  community convention. Fine.

- **Per-service repo `infra-<svc>/` dir**: clear and obvious. Adding
  a second service in another repo wouldn't pause for a layout
  decision.

- **SOPS config naming `.sops-<svc>.yaml`**: lets the legacy
  `.sops.yaml` coexist on main during the cutover. Good. After
  Stage 12 in the decommission playbook, rename to plain
  `.sops.yaml` and delete the legacy one.

- **GHA permissions blocks**: present in
  `nixos-deploy.yaml` (actions: write for cache purge) and
  `build-host-image.yaml`. Missing from `state-backup.yaml` and
  `tofu-apply.yaml` (both use the default permissions which is
  fine, but being explicit is a convention worth enforcing
  consistently — even if just `contents: read`).

## Platform/service seam (the headline)

The single bootstrap-time handoff (one CF R2 read+write token pasted
into the service repo's GitHub Environment) is the *only* manual
cross-repo step. Everything else flows via
`data.terraform_remote_state.platform`:

- VM IP and age recipient (drives runtime-secrets encryption)
- CI deploy SSH private key (drives every deploy)
- AWS DNS access keys (scoped via Route 53 normalized name
  conditions to the service's declared records)
- Runtime R2 token id + sha256(secret) (drives the app's S3 client
  and Litestream)
- Honeycomb backend + browser ingest keys (drive in-stack telemetry
  + Vite bundle)
- Zone ids, R2 bucket names, CF account id (re-exported for
  convenience)

The asymmetry is correct: things that are sensitive AND need to be
short-lived (R2 token rotation, Honeycomb key rotation) come from
state, no commits required. Things that are sensitive but stable
(ROCKET_SECRET_KEY, the platform's bootstrap age recipients) live in
encrypted-at-rest committed files.

The one wart is **S7 above**: the R2 runtime token is in the static
encrypted file rather than fetched per-deploy. That's the only
remaining inconsistency in an otherwise very clean model.

### Adding a second service: walkthrough

If I wanted to add a second service `widget-tracker` running on
sugar-glider:

1. `infra/tofu/credentials_delegations.tf` add a `widget = { ... }`
   entry. `tofu apply`. Mints IAM, R2 buckets, Honeycomb keys.
2. Capture the bootstrap token: `tofu output -json
   service_bootstrap_tokens | jq '.widget'`.
3. Paste two values into the new repo's GitHub Environment.
4. New repo: copy `infra-sillybus/` to `infra-widget/`, change the
   `sillybus` strings to `widget` (or templatize with a variable),
   point `dns.tf` at the right hostnames.
5. New repo: copy `.github/workflows/deploy-sillybus.yaml` to
   `deploy-widget.yaml`, change the service name, ports, image
   path.
6. New repo: write the widget's `docker-compose.nixos.yml`,
   `nginx/default.conf` if needed, `.sops-widget.yaml`.
7. Encrypt the widget's runtime-secrets file with `sops -e`.

That's it. No platform NixOS change. No new manual SSH key. No new
deploy user. The `/srv/*` sudo rule pattern accepts any new service
dir. The Traefik file provider auto-routes any new YAML file.

This is a meaningfully short list for "add a service to a platform"
and reflects the platform/service seam doing its job.

### Failure modes worth thinking about

- **Bootstrap token compromise.** It can read platform tfstate (all
  outputs, including sensitive ones for OTHER services). That's an
  acceptable blast radius for a solo-dev setup but would not be for
  a multi-tenant offering. Mitigation if/when needed:
  service-scoped state buckets (one bucket per service) with the
  bootstrap token narrowed to read only the platform's *public*
  outputs.
- **Platform state corruption.** Daily R2 snapshot to GHA artifact
  (`state-backup.yaml`) covers this. 60-day retention is generous.
  Recovery is `tofu state push -force` from the snapshot. Verify
  this restoration path works at least once before relying on it —
  the doc mentions it but nothing in the repo proves it.
- **CI deploy SSH key rotation.** `ci_deploy_ssh_key.tf:14-19`
  flags the deploy gap risk in the rotation comment. Worth turning
  into a script (`scripts/rotate-ci-deploy-key.sh`) that does the
  dual-key dance automatically rather than leaving it as a comment.

## Workflow currency

All workflows are up-to-date with their action versions and pinned to
SHAs where the action is sensitive (checkout, docker/setup-buildx,
docker/login, docker/build-push, dtolnay/rust-toolchain, etc.).
`cachix/install-nix-action@v27` is a tag rather than a SHA — that's a
weaker pin. Either accept it or pin.
